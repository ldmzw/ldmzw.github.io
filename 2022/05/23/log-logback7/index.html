日志组件Logback之Policy

发表于 2022-05-23 更新于 2022-06-20 分类于 java ， 日志组件 ， logback 阅读次数：

概述

在日志组件logback中，我们知道有Logger，Appender，Encoder，Layout，Filter等重要组件。日常使用中，我们很多时候都会使用RollingFileAppender来做日志文件滚动归档。那么日志文件是什么时候去执行滚动归档操作的,又是怎么样去归档的呢，这就不得不谈一谈logback中的RollingPolicy与TriggeringPolicy。TriggeringPolicy解决了when的问题，告诉系统什么时候去进行日志文件归档。RollingPolicy解决了how的问题，告诉系统日志文件应该怎么样去归档。

在logback中提供了很多触发策略，我们可根据提供的属性进行个性化配置。在进行日志文件滚动归档的时候，大体可以分为两种策略：文件大小和时间。

RollingPolicy

logback已经给使用者提供了几个常用的RollingPolicy，包括TimeBasedRollingPolicy、SizeAndTimeBasedRollingPolicy等。

TriggeringPolicy

TimeBasedRollingPolicy

根据时间进行日志文件归档应该是日常项目中使用最多的滚动策略。我们可以配置日志文件按月、按天或者按时进行归档。非常值得注意的是，TimeBasedRollingPolicy继承了RollingPolicyBase，同时也实现了TriggeringPolicy。因此，TimeBasedRollingPolicy既是RollingPolicy也是TriggeringPolicy。

从源码中我们不难看出，TimeBasedRollingPolicy实现TriggeringPolicy的具体执行还是通过配置的timeBasedFileNamingAndTriggeringPolicy来完成的。

配置项

属性名称 类型 描述
fileNamePattern String 日志文件名称，关键字%d必须包含，会根据文件名称中的%d来确定滚动周期。
%d后面可以自定义时间格式化，框架采用java.text.SimpleDateFormat来格式化，如果我们没有指定，那么默认为yyyy-MM-dd。
示例：/var/log/myapplication/info.%d{yyyy-MM-dd}.log。
我们除了可以自定义格式外，还可以自定义时区，采用java.util.TimeZone.getTimeZone(String)来获取时区，如果未配置或者配置不正确，则采用GMT。
示例：/var/log/myapplication/info.%d{yyyy-MM-dd, UTC}.log。
当然，我们的文件名称中也可以包含多个%d关键字。但是只能有一个%d作为主要关键字，用来确定滚动周期，其它的%d需要使用aux参数定义为辅助关键字。
示例：/var/log/%d{yyyy/MM, aux}/myapplication/info.%d{yyyy-MM-dd}.log。
maxHistory int 日志文件最大归档数量，当归档数量超过配置数量时，会异步删除最早的日志文件归档。默认为0，即不删除任何归档文件
totalSizeCap int 日志文件总大小。我们可以在配置的时候配置单位，例如：500，500KB，500MB，500GB。如果我们未配置单位，则默认为 byte
cleanHistoryOnStart boolean 默认为false，如果设置为true，则将在appender启动的时候删除所有的历史日志文件归档

注意：

在日志文件进行归档时，可以配置fileNamePattern以.gz或者.zip结尾来开启自动压缩。 以logFile.%d.gz为例

  • 如果appender中没有设置file属性，那么当天的日志会被记录到logFile.2018-07-13中，当到达24点时，logFile.2018-07-13会被压缩归档至logFile.2018-07-13.gz，同时日志输出将被重新定向到logFile.2018-07-14中
  • 如果appender中设置了file属性，如lm.txt。当天的日志会被记录到lm.txt中，当到达24点时，lm.txt将会被压缩重命名为logFile.2018-07-13.gz。同时新的文件lm.txt会被创建并记录新的日志。

使用

SizeAndTimeBasedRollingPolicy

SizeAndTimeBasedRollingPolicy继承TimeBasedRollingPolicy，唯一不同的是它们中的timeBasedFileNamingAndTriggeringPolicy。

RollingPolicy TriggeringPollicy
TimeBasedRollingPolicy new DefaultTimeBasedFileNamingAndTriggeringPolicy()
SizeAndTimeBasedRollingPolicy new SizeAndTimeBasedFNATP(Usage.EMBEDDED)

配置项

属性名称 类型 描述
maxFileSize int 日志文件总大小。我们可以在配置的时候配置单位，例如：500，500KB，500MB，500GB。如果我们未配置单位，则默认为 byte

使用

注意：

在fileNamePattern中，除了%d关键字，还有%i关键字。%d与%i必须同时存在。当在一个时间周期内，如果日志文件达到了maxFileSize，日志文件将会从0开始，并顺序增长进行归档。
以上配置中，如在2018-07-13时，如果logFile.log达到了100MB，那么日志文件会进行压缩归档logFile.2018-07-13.0.gz，logFile.2018-07-13.1.gz，logFile.2018-07-13.2.gz。。。

上面的示例配置使用了TimeBasedRollingPolicy和SizeAndTimeBasedFNATP，与使用SizeAndTimeBasedRollingPolicy实现效果是一样的。在1.1.7版本之后，logback推荐我们直接使用SizeAndTimeBasedRollingPolicy。如果我们使用的是TimeBasedRollingPolicy和SizeAndTimeBasedFNATP，日志框架也能正常运行，但是会打印如下提示。

FixedWindowRollingPolicy

FixedWindowRollingPolicy会保留固定数量的日志文件归档，它与前面我们所说的TimeBasedRollingPolicy与SizeAndTimeBasedRollingPolicy不同，它只实现了RollingPolicy，并没有实现TriggeringPolicy，所以它的使用需要配合其它TriggeringPolicy一起使用才行。

配置项

属性名称 类型 描述
minIndex int 窗口的最小边界，默认为1
maxIndex int 窗口的最大边界，默认为7
fileNamePattern String 日志文件名称，关键字%i必须包含

注意：

如果我们没有特别指定，窗口最大值默认为20，如果窗口值maxIndex - minIndex > 20时，会自动将窗口调到20

使用

注意：

在使用FixedWindowRollingPolicy时，Appender的file属性必须设置，虽然rollingPolicy中设置有fileNamePattern。这点与前面讲到的TimeBasedRollingPolicy与SizeAndTimeBasedRollingPolicy也不同。

SizeBasedTriggeringPolicy

SizeBasedTriggeringPolicy根据日志文件大小来触发滚动，它的使用十分的简单，可以参照上面的FixedWindowRollingPolicy使用，只有一个参数maxFileSize，默认是10M。

配置项

属性名称 类型 描述
maxFileSize int 日志文件的大小，默认为10M

TimeBasedFileNamingAndTriggeringPolicy

TimeBasedFileNamingAndTriggeringPolicy比较特别，它需要传入一个TimeBasedRollingPolicy，并且根据TimeBasedRollingPolicy中fileNamePattern的%d来确定滚动触发策略。从命名我们也能看出，它主要是用于TimeBasedRollingPolicy中的timeBasedFileNamingAndTriggeringPolicy属性。

logback框架中给我们设置了两种TimeBasedFileNamingAndTriggeringPolicy：

  • DefaultTimeBasedFileNamingAndTriggeringPolicy：默认的日志滚动触发策略。TimeBasedRollingPolicy如果未指定timeBasedFileNamingAndTriggeringPolicy，默认使用的就是此策略，根据时间进行滚动。
  • SizeAndTimeBasedFNATP： 与默认的策略不同，除了使用了TimeBasedRollingPolicy中的时间，还需要设置日志文件大小，同时根据时间与日志文件大小进行滚动。SizeAndTimeBasedRollingPolicy的实现就就是该触发策略来完成的。

结语

日常使用中，大多情况下我们都会以时间进行归档，框架预置的RollingPolicy与TriggeringPolicy已经满足我们的大部分需求。当然，我们也可以去自定义我们的RollingPolicy与TriggeringPolicy，实现我们自己的一些业务逻辑。
